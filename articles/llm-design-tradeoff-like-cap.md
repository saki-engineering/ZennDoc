分散システムを学ぶ人なら一度は「CAP定理」という言葉を耳にしたことがあるでしょう。
Consistency(一貫性)・Availability(可用性)・Partition tolerance(分断耐性)の3つを同時に満たすシステムは作れない、という分散システムのよく知られた性質をこの定理は表しています。
システム設計を行うアーキテクトはどこかでこのトレードオフに直面し、どれを優先しどれを落とすのかを選択する必要に迫られることになります。

このような設計上のトレードオフは分散システムに限ることなく、技術を用いてモノを作る際には何かしら発生するものです。
今回はその中でも、昨今ブームのLLMアプリケーションにも普遍的な設計上のトレードオフ - CAP定理があるのではないか、という話をしようと思います。




# 分散システム設計におけるトレードオフ - CAP定理

まずは前提として、分散システムにおけるCAP定理を簡単に振り返ります。

CAP定理とは、分散データストアが以下の3つの保証を**同時にすべて満たすことはできない**という定理です。

- **C（Consistency / 一貫性）**: どのノードに問い合わせても、常に最新かつ同じデータが返る
- **A（Availability / 可用性）**: すべてのリクエストに対して、必ずなんらかのレスポンスが返る
- **P（Partition tolerance / 分断耐性）**: ネットワーク分断が発生しても、システムとして機能し続ける

複数インスタンスを用いて構成・複数拠点で使われるようなシステムであれば、分断耐性は捨てることはできません。昨今の規模のシステムであれば、Pは必ず重要視して確保する必要があります。
その上でCとAのどちらを優先するか、現実的にはCとAは0か100かの選択ではないので、どの程度のCとAを担保するのかという選択が現代のアーキテクチャの本質となります。

一貫性であれば、

- 最終的に整合が取れるのであれば一時的な不整合を許す = 結果整合か、常にデータ整合を要求する = 強整合か
- ACID特性を持つデータベースであれば、どのレベルのトランザクション分離レベルを採用するか

といった選択をユースケースに合わせてすることになりますし、可用性であれば、システム特性に応じてSLOを適切に策定・調整することになります。





# LLMアプリケーションにおけるトレードオフ - CAP定理

さて、今度はLLMを用いたアプリケーションについて考えてみましょう。筆者も最近はよくこの手のアプリケーションを作るための設計を行なっています。
何度も何度も設計を行うたびに、上述の分散システムのCAP定理と驚くほど似たトレードオフに毎回直面するということに気づきました。

- **C（Consistency / 一貫性）**: 全ユーザーリクエストに対して常に同じモデルを使うことで、出力品質を安定させる
- **A（Availability / 可用性）**: TPM（Tokens Per Minute）・RPM（Requests Per Minute）などのクオータ制限に引っかからず、LLMがリクエストに応答し続けられる
- **P（Price / コスト）**: 安価な値段でLLMを利用することができる

筆者はこの3つが分散システムのCAPと同様に同時に満たすことができないと考えています。以下でそれぞれの組み合わせを見ていきましょう。

## CAを選ぶ場合（Pを捨てる）

一貫した高品質な出力を保ちながらも、可用性も確保したいということを考えます。
その場合、**システムで利用するモデルを1つに絞りそこにチューニングリソースを全振りした上で、当該モデルの十分なクオータを確保する** という戦略を採ることになります。

たとえばAmazon Bedrockでは一部のモデルに対してReserved Tierという料金体系を提供しており、これを利用することによってある一定のTPMを予約することができます。
Azure OpenAIでもPTUを、Google CloudのVertex AIでもProvisioned Throughputを購入することによって、モデルの推論容量を確実に確保することができます。

ただし、これらリソースの予約を行う料金体系を利用するには相応のコストがかかります。
筆者の体感ですが、プロビジョンドスループットのトークン単価は従量課金より割高な設定であることが多く、またピーク時のために確保したキャパシティは閑散時には無駄になります。

したがってこれは、**CとAは手に入るが、Pが犠牲になる**という構図です。

## CPを選ぶ場合（Aを捨てる）

出力品質は維持したいしかつコストも抑えたいということを考えます。
その場合、採る戦略は **性能の良いモデルを従量課金で1つ使い続ける** というものになります。

しかし、アプリのリクエストがある程度の規模になると、TPMやRPMのクオータに引っかかるようになります。
するとモデルからは`429 Too Many Requests` が返り始め、それを受けたアプリケーション側ではリトライやバックオフ試行を行うことになります。
結果、エンドユーザー見えとしてはレスポンスが遅延・失敗するような体験になってしまいます。

コストは抑えられているかもしれませんが可用性はイマイチ、**CとPは手に入るが、Aが犠牲になる**という構図です。

## APを選ぶ場合（Cを捨てる）

可用性を確保しつつ安くモデルを利用したいという場合、採る戦略は **複数のモデルを組み込んで、そのとき利用可能なモデルを分散利用する** というものになります。
例えば、

- 通常はとあるモデルAを従量課金で使いつつ、そのモデルがTPM/RPM超過で異常応答を返し始めたら異なるモデルBに切り替える
- リクエストの内容によって異なるモデルを使い分ける

という設計です。安価な従量課金・安価なモデルを複数利用することで可用性を担保しています。

例えばAzureのAPI Managementでは、バックエンドにあるモデルの一つが429エラーを返し始めたら別モデルにフォールバックさせるような[AI Gateway](https://learn.microsoft.com/ja-jp/azure/api-management/genai-gateway-capabilities)の構築を行うことができます。
Cloudflare AI Gatewayにも[Fallback](https://developers.cloudflare.com/ai-gateway/configuration/fallbacks/)という同様の機能が存在します。

ただし、このアプローチでは**モデルの出力品質が一貫しなくなります**。
モデルAでは上手くいっていた処理がモデルBでは崩れる、複数個の性質の違うモデルを同時にTuning・メンテナンスしないといけないといった課題が生じます。
**AとPは手に入るが、Cが犠牲になる**という構図です。





# まとめ

こうして整理すると、分散システムのCAP定理とまったく同じ構造が見えてきます。

- **C**onsistency = 利用モデルの一貫性
- **A**vailablity = LLMの可用性
- **P**rice = LLMの利用料金

**この3つはトレードオフであり、同時に成立させるような設計は不可能です。**

ただし、分散システムのCAP定理が「Pは死守」となる傾向が強いのに対し、LLMアプリケーションにおいてはCAPどれもが調整の余地が残っています。

- 最初は一つのモデルで安く始め(=CP重視)、サービスが盛り上がってきたら複数モデルの利用(=Cを落とす)やプロビジョンドスループット(=Pを落とす)を検討してAを向上させる
- ユーザーにモデルを選択させるなど複数モデル利用を前提としたプロダクト設計にすることで、最初からCを捨てる
- 高いモデル品質・高い可用性(=CA重視)でユーザーを確保したのち、利用頻度が低いライトユーザー向けには安価なモデルプランを提供することでPを下げる

といった様々な調整策を打つことが、ビジネスサイドやプロダクトオーナーと密に連携・コミュニケーションをとることで可能になります。

これからのシステムアーキテクトは、このような優先度・トレードオフの調整を**開発だけではなく、ビジネスサイドを巻き込んで行い、設計に落とし込む能力**が必要になるのではないでしょうか。
その設計判断を言語化し議論しやすくするにあたって、このLLMのCAP定理という枠組みが役に立つようでしたら嬉しいです。
