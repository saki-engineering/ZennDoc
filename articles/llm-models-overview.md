# この記事について

分散システムを学ぶ人なら一度は耳にする「CAP定理」。Consistency（一貫性）・Availability（可用性）・Partition tolerance（分断耐性）の3つを同時に満たすシステムは作れない、というあの定理です。

LLMを使ったアプリケーション開発を進めていくなかで、これに似た構造のトレードオフに直面していることに気づきました。本記事では、このトレードオフを「LLMアプリのCAP定理」と名付けて整理し、

- LLMアプリ開発でなぜこの3つが同時に成立しないのか
- それぞれどのようなトレードオフが生まれるのか

という観点から、アーキテクチャ設計の考え方を提言することを目的としています。

## 使用する環境・バージョン

特定のサービス・バージョンには依存しません。OpenAI API・Anthropic API・Amazon Bedrock など、クオータ制限を持つLLMサービス全般に該当します。

## 読者に要求する前提知識

- LLMのAPIを使ったアプリケーション開発の基礎
- 分散システムのCAP定理の概要（知らなくても読めますが、知っていると理解が深まります）

# 分散システムのCAP定理（おさらい）

まずは前提として、分散システムにおけるCAP定理を簡単に振り返ります。

CAP定理とは、分散データストアが以下の3つの保証を**同時にすべて満たすことはできない**という定理です。

- **C（Consistency / 一貫性）**: どのノードに問い合わせても、常に最新かつ同じデータが返る
- **A（Availability / 可用性）**: すべてのリクエストに対して、必ずなんらかのレスポンスが返る
- **P（Partition tolerance / 分断耐性）**: ネットワーク分断が発生しても、システムとして機能し続ける

現実のネットワークは必ず分断（Partition）が起こりうるため、PはほぼMustです。その上でCとAのどちらを優先するか、という選択がアーキテクチャの本質となります。

「全部取れればいいのに」と思いますよね。でも数学的に証明されているこの定理は、設計者に厳しい現実を突きつけます。

# LLMアプリにも同じ構造がある

LLMを使ったアプリケーションの設計を進めていると、CAP定理と驚くほど似たトレードオフに直面します。

私はこれを **LLMアプリのCAP定理** と呼んでいます。構成要素は以下の3つです。

- **C（Consistency / 一貫性）**: 常に同じモデルを使い、出力品質が安定している
- **A（Availability / 可用性）**: TPM（Tokens Per Minute）・RPM（Requests Per Minute）などのクオータ制限に引っかからず、リクエストに応答し続けられる
- **P（Price / コスト）**: モデルの利用コストが最適化されている

筆者はこの3つが分散システムのCAPと同様に、同時に満たすことができないと考えています。以下で、それぞれの組み合わせを見ていきましょう。

---

# 3つのトレードオフを深掘りする

## CAを選ぶ場合（Pを捨てる）

一貫した高品質な出力を保ちながら、可用性も確保したい。その場合、採る戦略は **高性能なモデルを1つに絞り、プロビジョンドスループットなどで十分なクオータを確保する** というものです。

たとえばAmazon Bedrockでは「プロビジョンドスループット」を購入することで、より高いTPMを確保できます。OpenAIでも高いTierに移行すれば、より大きなレート制限が利用可能です。

ただし、これには相応のコストがかかります。プロビジョンドスループットは従量課金より割高であり、ピーク時のために確保したキャパシティは閑散時には無駄になります。**CとAは手に入るが、Pが犠牲になる**という構図です。

## CPを選ぶ場合（Aを捨てる）

品質は維持したい、かつコストも抑えたい。その場合、採る戦略は **性能の良いモデルを1つ使い続けるが、プロビジョンドスループットは購入しない** というものです。

しかし、アプリのリクエストがある程度の規模になると、TPMやRPMのクオータに引っかかるようになります。`429 Too Many Requests` が返り始め、リトライやバックオフが増え、ユーザーへのレスポンスが遅延・失敗します。

コストは抑えられているかもしれませんが、**可用性が犠牲になる**という構図です。

## APを選ぶ場合（Cを捨てる）

可用性を確保しつつコストも最適化したい。その場合、採る戦略は **複数のモデルやプロバイダーを分散して利用する** というものです。

たとえば、通常はコストの低いモデルを使いつつ、一方が詰まったらもう片方に切り替える、あるいはリクエストの内容によってモデルをルーティングする、という設計です。クオータの心配は減り、コストも抑えられます。

ただし、このアプローチでは**モデルの出力品質が一貫しなくなります**。モデルAでは上手くいっていた処理がモデルBでは崩れる、プロンプトの微妙なニュアンスへの対応が変わる、といった問題が生じます。APを選ぶことで、**一貫性（C）が犠牲になる**というわけです。

## 3つ同時には無理

こうして整理すると、分散システムのCAP定理とまったく同じ構造が見えてきます。

- Cを確保しようとすると → 1つのモデルに絞る → クオータが詰まる（A低下）か、コストがかかる（P低下）
- Aを確保しようとすると → モデルを分散させる → 品質がバラつく（C低下）か、コストがかかる（P低下）
- Pを最適化しようとすると → 安いモデルを1本に絞る → クオータが詰まる（A低下）か、分散させて品質がバラつく（C低下）

**どこかで必ず何かを諦める。3つ同時には成立しません。**

:::message
分散システムのCAPと異なる点として、LLMアプリのCAPには「緩和策」がないわけではありません。モデルの進化・価格競争・クオータ引き上げ交渉などによってトレードオフの幅は変化します。ただし、ある時点での設計判断においては、常に何かを優先・犠牲にする選択が伴います。
:::

# まとめ

本記事で提言した「LLMアプリのCAP定理」をまとめます。

- C（一貫性）・A（可用性）・P（コスト）の3つは、LLMアプリ設計においてトレードオフの関係にある
    - CA → プロビジョンドスループット等でクオータ確保 → コスト増（Pが犠牲）
    - CP → 良いモデルを1本に絞る → クオータ枯渇で可用性低下（Aが犠牲）
    - AP → 複数モデルを分散利用 → 品質一貫性の低下（Cが犠牲）
- 一貫性を保つには単一モデルへの集約が必要で、それはクオータかコストに跳ね返る。可用性のために分散させると一貫性が崩れる

もちろん、「LLMアプリのCAP定理」はあくまで筆者の整理であり、数学的に証明されたものではありません。ただ、LLMアプリの設計では「何を優先するか」を意識的に決めることが必要であり、こうした枠組みで語ることで設計の議論がしやすくなるのではないかと思っています。

分散システムのCAP定理が「3つのうちどれを諦めるか」という設計判断を言語化したように、LLMアプリにおいても同様の枠組みが役に立つのではないでしょうか。
