# この記事について
「とりあえずClaude Codeを入れてみたけど、なんか使いこなせていない気がする」「あの機能の存在は知っているけど、自分には関係ないと思っていた」という方はいませんか？

本記事では、Claude Codeを使いはじめた人が日々の作業の中で感じるよくある「つまずき」や「不満」を出発点に、それを解消するClaude Codeの機能を紹介していきます。
- Claude.aiチャットとClaude Codeのagentモードは何が違うの？
- CLAUDE.md、Hook、MCP、Skillなど設定項目が多すぎてどこから手をつければいいかわからない

という部分を、「こういう不満があるなら、この機能を使いましょう」という形で解説し、Claude Codeを業務に本当に活かせる状態を目指すことを目的としています。

:::message
本記事では各機能の詳細な使い方よりも、「どんな課題を解決するための機能か」という視点での紹介に留めます。詳細な設定方法はClaude Code公式ドキュメントをご参照ください。
:::

## 使用する環境・バージョン
```bash
$ claude -v
2.1.76 (Claude Code)
```

# Claude Code 使いこなしQ&A集

## Claude.aiのチャットだけでは不十分かな？
Claude.aiのWebサイトやアプリを開いて、「(コードをコピペして)こういうコードを書いているんだけど、どこがおかしいか候補を出して」「XXXの機能を実装するとしたらどういう方式が考えられる？」とアドバイスを求めてみるような使い方もできますが、Claude.aiのチャットはあくまで「会話」が中心です。
Claude.aiを使っているだけだと、価値のある会話をするために必要なファイルの情報をコピペして渡し、Claudeの回答を元にコードを書き、コマンドを実行するといった「作業」の部分は人間が手を動かす必要があります。

Claude Codeではその作業まで丸ごと任せることができます。
エディタ上でファイルを開いているだけで(なんならVSCode上開いているレポジトリ全体の内容でさえも)、そのファイルの内容をコンテキストとしてClaudeに渡すことができます。
人手でファイルの内容をコピペをする必要は一切なく、チャット欄には「今この瞬間やって欲しい作業指示・聞きたい質問の内容」だけを書くだけで済みます。

また、関連ファイルを楽に読み込めるだけではなく、コードを修正し、テストコマンドを実行するといった開発作業まで実行させることができます。
なんなら「XXXXというバグがあるけど解決して」と一言伝えるだけで、そのバグの原因がありそうなファイルを探し見つけて読み込み、それを編集してバグを修正し、本当に直ったかどうかテスト実行して確認する、直っていないようであればもう一度修正→テスト実行を試みる...といった作業をしてくれます。

Claude.aiのチャットが「アドバイスをもらう場」なら、Claude Codeモードは「そのまま開発作業をまるっとClaudeに任せて代行してもらう場」というイメージです。


## 毎回同じ指示を出している気がする...チーム固有のルールを覚えてほしいんだけど、できる？

- うちのプロジェクトではコミットメッセージは英語で書く
- テストコマンドは`make test`

といったローカルルールを毎回会話の冒頭で説明するのはめんどくさいですし非効率です。
そこで使えるのが`CLAUDE.md`です。
このファイルをプロジェクトのルートにおいておくとClaude Codeがセッション開始時に必ず読み込むので、そこにテストコマンドやブランチ命名規則、コーディングスタイルなどのプロジェクト固有のルールをここに書いておくことで、毎回説明する手間を省くことができます。

```markdown
## テスト
- テストコマンド: `make test`

## ブランチ命名ルール
- feature/[issue番号]-[概要]

## その他ルール
- コミットメッセージは英語で書くこと
```

## そのチーム固有のルールとやらをわざわざ言語化するのもめんどくさいズボラなんですけど、なんとかなりますか？

その気持ちはほんと〜〜〜によくわかります。
`CLAUDE.md`に書くルールを一から考えて言語化するのも面倒なんですよね。そんなときは`/init`コマンドが使えます。

Claude Codeが現在のリポジトリを読み込み、コードスタイルや既存のスクリプトなどを解析して、CLAUDE.mdの雛形を自動生成してくれます。生成されたものを確認・調整するだけでよいので、白紙から書くよりずっと楽です。

:::message
どんどん開発を進めて後述するRule / Commands / Skills / Agentを導入すると、わざわざ`CLAUDE.md`に書かなくても向こうに書けばいいなという内容が出てきてリファクタされていくかと思いますが、初手はこんなもんで十分です。
:::

## Claude Codeが作業をするたびに、一挙手一投足いちいち許可を求められるのが煩わしいんだけど...

Claude Codeはファイルを読んだり編集したり、コマンドを実行したりするたびに「この操作をしてもいいですか？許可しますか？」と聞いてきてくれます。
Vibe Codingに慣れるまでは、何をやらかすかわからない余所者のClaudeがやることを逐一把握できるので安心感がありますが、作業を繰り返すうちにこの確認が煩わしくなってきます。
特にファイルの読み込みやWeb検索など「流石にこれはやってくれないと作業にならんだろ」というものは、もう許可なしでよしなに実行してくれよという気持ちになるでしょう。

このような許可を出さなくても自動で実行して欲しい操作は、レポジトリ内の`./claude/settings.json`の`permission`に書いておくことでいちいち確認なしに実行させることができます。WebSearchやWebFetchといった情報収集系の操作は許可しておいても問題ない場面が多いでしょう。

```json:.claude/settings.json
{
  "permissions": {
    "allow": [
      "Bash(go test:*)",
      "Bash(go generate:*)",
      "Bash(go vet:*)",
      "Bash(mkdir:*)",
      "Bash(touch:*)",
      "Bash(ls:*)",
      "Bash(grep:*)"
    ]
  }
}
```

また、`settings.json`の記述フォーマットをいちいち自分で調べて書くのもめんどくさいという方は、`/permissions`コマンドを使って対話的に許可リストを管理することもできます。

:::message
セキュリティ面で懸念があるような危ない操作をClaude Codeが勝手に実行しないように、許可リストに入れるコマンドは適切に精査・選択する必要があります。
必要であれば許可リストだけではなく拒否リストの方にも危ないコマンドを記述してガードレールをきちんと構築しておくことが実務では重要です。
```json:.claude/settings.json
{
  "permissions": {
    "allow": [
      /*略*/
    ],
    "deny": [
      "Bash(sudo:*)" // sudoコマンドを拒否する
    ]
  }
}
```
:::


## Claude.mdに書いたルールを最近Claudeが守ってくれなくなっちゃったのだけどどう躾けたらいいですか？
例えば、「`CLAUDE.md`に『コードを編集したらXXXのフォーマッタをかけて成形すること』と書いたのに、いざClaudeに書かせたコードを見たらフォーマッタがかかっていなかった」ということがあります。
これはClaudeに渡す各種プロンプトが、本質的にはただの「お願い」であることに起因します。Claudeはコンテキストや状況に応じてプロンプトの解釈を変えることがあり、100%その通りになる保証はないのです。

その代わり、強制的に実行・適用させたいルールには**Hook**を使いましょう。
Hookは特定の操作（ファイル編集後、コマンド実行前など）に紐づけてスクリプトを実行する仕組みです。これを利用することで、「Claudeがファイルを編集したら必ずフォーマッターを実行する」といった処理を、会話ではなく仕組みとして担保することができます。

```json:.claude/settings.json
// (例) ファイル編集後にGoのformatterをかけるhook
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "gofmt -w $CLAUDE_TOOL_RESULT_FILE_PATH"
          }
        ]
      }
    ]
  }
}
```

## Claude Codeに作業を任せて放置しておいたら、処理実行許可を求めるだいぶ序盤のところで止まってました。やっぱり完全放置はできなくてずっと画面を見張ってないといけないんですか？

Claudeに作業を任せて自分は他のことをしていて、後から様子を見にきたらClaudeが入力待ち状態で止まっていたという経験はないでしょうか。
せっかくClaudeと自分で作業を分担できるようになって並列に動けるようになりそうだというのに、このようなことが起きると台無しですよね。

この問題もHookで解決できます。離席中でも気づくことができるように、「ユーザーの入力が必要な状態になったら音を鳴らす」という処理をHookで登録しておけばいいのです。

```json:~/.claude/settings.json
// (例) Macの場合
{
  "hooks": {
    "PermissionRequest": [
      {
        "matcher": "*",
        "hooks": [
          {
            "type": "command",
            "command": "afplay /System/Library/Sounds/Glass.aiff"
          }
        ]
      }
    ],
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "afplay /System/Library/Sounds/Glass.aiff"
          }
        ]
      }
    ]
  }
}

```

## 前の会話に引きずられておかしな動きをし出したんだけど....

作業を続けていくと、前の発言で失敗したアプローチがコンテキストに残り、それに引きづられて同じ間違いを繰り返したり、的外れな修正を始めたりすることがあります。
Claudeが間違える→修正する→また間違えるというループに陥っているなら、それはコンテキストが失敗したアプローチで汚染されているサインです。
失敗したアプローチがコンテキストの中に残っている状態だとその後もそれを背景情報として読み取って作業を実行しようとするため、当然うまくいきません。

こういうときは躊躇せず`/clear`で会話履歴をリセットするか新しいセッションを開始しましょう。そうするとまっさらなコンテキストで作業することができます。

## 一つのセッションで話したいことが多すぎてそれだけでコンテキストが溢れてしまうんだけど、どうやって会話を継続すればいい？

会話が長くなってコンテキストウィンドウの上限に近づいてきたときにはどうすればいいでしょう。
`/clear`コマンドを使ったり新しいセッションを立てたりするとコンテキストがリセットされてしまうので、できればそれはやりたくないというパターンもあるかと思います。

そういうときは`/compact`コマンドを使いましょう。
`/compact`は会話履歴を要約してコンテキストをコンパクトにまとめ直します。全ての詳細を保持するわけではありませんが、作業の文脈は維持しながらコンテキスト使用量を減らすことができるため、`/clear`より穏やかなリセット手段として活用できます。

## これから色々試行錯誤をしてコンテキストを汚すからその前に今の状態を保存して後で戻れるようにしたいんだけど、一時セーブとかできる？

「ここから実験的なアプローチを試してみたいけど、失敗したら今の状態に戻りたい」というときには、**rewind**と**fork**が使えます。

`rewind`は直近1つの操作を元に戻します。今の失敗したアプローチをなかったことにして続行したいというときには有効です。`git reset`のような操作です。
`fork`は現在のセッションを「親」として、その続きを引き継いだ新しいセッションを作ります。新しいセッションで会話を継続しても元のセッションには変更が加わらないため、forkしたセッションでうまくいかなくても、その情報を持たない親セッションに戻って別アプローチを試すということが可能です。
gitのブランチのセッション版と思えばわかりやすいかもしれません。

## Claude Codeにちょっと大きい設計・実装作業を任せてるのだけど、Agentモードで逐一ペアプロしていると軌道修正ばかりでなかなかゴールに辿りつかない...

Claude Codeと会話しながら作業をしていると「今Claudeがざっくり何を目的として行動しているのか」「最終的にどこに着地するのか」が見えないまま、どんどん変更が加えられてしまうことがあります。
1回ずつの変更だけ見ていても全体像が把握できないから、最初に方針や全体像を見せてほしいという気持ちになる場面もあるかと思います。

そういうときはClaude Codeのモードを変更しましょう。
普段会話しながら一回一回許可を出して変更を加えていくのは**Ask before edits**と呼ばれるモードです。
そうではなく、全体像を見てから実装に取り掛かりたいという場合には**Planモード**が便利です。
実装に入る前にClaudeが「こういう方針で進めます」という計画や、計画が承認された場合のTODOリストを提示してくれるので、人間がそれを確認してからGOサインを出すことができます。
また、Planで承認した後の実装フェーズでは**Edit automatically**と呼ばれるモードに変更されるので、Planで大まかに方針合意が取れている個別の操作ごとに許可を求めることなく一気に作業を進めてくれます。

このようにモードの切り替えを活用することで、全体像を確認してから実装は高速にという流れが実現できます。
ちょっと大きめな機能実装や、方針策定・設計から始めるような実装タスクの場合は、このPlanモードが特に有効です。


## 大きな作業をしていると、1つのセッションで使えるコンテキストがすぐに枯渇してしまうようになった。何かいいコンテキスト節約方法はある？

大きな実装・設計タスクをClaudeに任せていると、試行錯誤の過程でコンテキストウィンドウがすぐに一杯になってしまうことがあります。
`/compact`を用いて処理継続していたとしても、コンテキストが圧縮されることでどんどん最初に頼んだタスクの目的や内容が薄れて忘れられていく...ということを感じるかもしれません。

こんなときは**SubAgent**を活用しましょう。
例えばとある機能設計をClaude Codeにさせるときに、その作業の過程で実装の参考になる先行事例や、実装で利用するライブラリの詳細な仕様を調べる必要が出てくるかもしれません。
こういうときは、その途中の子タスクであるリサーチ作業をSubAgentにやらせると、メインのセッションとは別セッションでその子タスクの作業を動かすことができます。
SubAgentの大きなメリットは、セッションが別であるため元の作業場所のコンテキストを一切汚すことなく作業させることができるという点です。試行錯誤の過程はSubAgentに押し付けて、最終的な結論だけ元のセッションに持ち帰らせる、という使い方でコンテキストを節約します。

このような別セッションへの切り出しを人間がやろうとすると、手動で別ウィンドウを開いてリサーチして、結果をコピーして元のセッションに貼り付けるといった作業が必要になりますが、その一連の作業ごとSubAgentに任せられる・なんならSubAgentを起動して別セッションで作業させた方がいいのかという判断までClaude Codeに任せられるのも魅力です。

WebSearchなどClaude Codeでデフォルトで用意されているSubAgentも多々ありますが、自作したい場合には`/agent`コマンドで作成できます。やりたいことの説明（description）を簡単に記述してコマンドを実行し、あとは使用するツールの設定などの質問に対話的に答えるだけで、簡単にカスタムサブエージェントを構築してくれます。
カスタムエージェントの内容は`.claude/agent`直下に格納されています。

## SubAgentを使ってもまだコンテキストが足りない...

ここまでClaude Codeを使いこなせるようになっているのなら、もしかして`CLAUDE.md`が肥大化してきていないでしょうか。
プロジェクトのREADMEと見分けがつかないくらい長くなっているなら、**rules**の活用を検討しましょう。

`./claude/rules`ディレクトリ直下に今まで`CLAUDE.md`に書いていたようなプロジェクト固有の指示を、複数のMarkdownファイルに分割して記述することができます。
```
.
├── CLAUDE.md
└── .claude/
    └── rules/
        ├── architecture.md
        ├── coding-rule.md
        └── etc...
```

:::message
CLAUDE.md内で`@file_name`と書くことでも、詳細な指示内容を別ファイル参照の形で逃すこともできます。しかし、`./claude/rules`直下に配置することで、そのような明示的指定なしでも別ファイルのコンテキスト読み込みを実現することができてお手軽です。
:::

また、rules特有の機能として`path`があります。以下のようにマークダウンファイル冒頭に「このruleを読み込んで欲しいのはどのようなファイルを触るときか」というパスを指定することで、本当に必要なときだけ当該ルールがコンテキストとして読み込まれるようになります。
この「必要になるまで読み込まれない」という仕様を利用することによってコンテキストを節約できます。
```markdown
---
paths: src/**/*.go
---
```

## エディタでファイルをいじる・ターミナルでコマンドを実行する以外の操作もClaudeに任せたいけど、できる？

自分の開発がだんだんClaude Code中心で回ってくるようになると、その作業内容のサマリをSlackへメッセージ投稿したり、作成したUIの出来をチェックするためにブラウザ上でスクリーンショットを撮ったりといった、エディタ上でファイルを編集する以外の作業もClaude Code起点でしたくなることもあるかもしれません。
そういう場合は**MCP（Model Context Protocol）**を活用しましょう。

MCPとは、外部サービスとClaude Codeを連携して繋ぐインターフェースプロトコルです。
例えば以下の`claude mcp add`コマンドを用いてplaywright MCPをインストールすることによって、playwrightを用いてスクリーンショットを撮ったりブラウザを操作したりといったtoolsをClaude Codeに導入することができます。
```bash
claude mcp add playwright -s project -- npx -y @playwright/mcp@latest
```

MCPは後述するpluginを用いてインストールすることもできます。
以下のように`/plugin`コマンドを用いてSlack MCPをインストールすることによって、Slackのチャネル検索・読み取り・メッセージ投稿といったtoolsをClaude Codeで利用できるようになります。
```claude
/plugin install slack
```

Claude Codeに追加したMCPの設定は`~/.claude.json`に記録されます（Claude Desktopの設定とは別管理です）。
```json
// (例) AWSのMCPサーバーを入れている場合
{
  "mcpServers": {
    "awslabs.core-mcp-server": {
      "command": "uvx",
      "args": [
        "awslabs.core-mcp-server@latest"
      ],
      "env": {
        "FASTMCP_LOG_LEVEL": "ERROR"
      }
    }
  }
}

```

:::message
MCPを使って外部ツールを操作するのとターミナルにインストールされたCLIコマンドを用いて外部ツールを操作するのは何が違うのだという話があります。
昨今[MCP is dead. Long live the CLI](https://ejholmes.github.io/2026/02/28/mcp-is-dead-long-live-the-cli.html)という記事が出たのも記憶に新しいでしょう。
ただ、MCPにはMCPの良さが、CLIにはCLIの良さがあります。
MCPはtoolsという形で外部ツールの操作がLLMにとって扱いやすい形で定義されているのに対して、トークン使用量が嵩むという側面があります。CLIは自分がやりたい操作をするためにはどのような引数を指定してコマンド実行すればいいのだろうか、という慣れが生まれるまでの大変さはありますが、コマンド実行という身軽さ故にトークン使用量は少なく済みます。どちらが優れているという話ではありません。
CLIで困らないものはCLIを使う、複雑な操作をしたいのであればMCPを使うなど、自分がやりたいことを精度高く実現できる方式を用途に応じて都度選択していくのがよいのかなと思います。
:::

## Claude CodeがWeb検索・WebFetchするときのやり方が非効率な気がする。何かいいチューニング方法はある？

MCPはツールを操作するだけでなく、情報を参照するためにも使えます。
例えば**context7**のMCPを使うと、ライブラリの公式ドキュメントを高精度で参照できます。Web検索を用いてももちろん必要な情報を拾うことはできますが、context7のようなドキュメント特化のMCPを用いることで、より限られた正確なソースの中から情報を取得しClaudeに渡すことができます。
もっと身近なところだとAWSも多数のMCPを公式提供しており、例えばAWS公式ドキュメントの内容をClaude Codeに参照させるような使い方をすることができます。


## MCPを増やしたら作業の精度が下がった気がする...

MCPは便利ですが、むやみやたらに追加していくとClaudeが大量にあるMCPツールの中からどれを使えばいいか判断に迷うようになり、かえって精度低下につながることがあります。
これを防ぐためには「今必要な分のtoolsのみを追加してClaudeの選択肢に入れる」という取捨選択が必要になります。

MCPに限った話ではないですが、Claude Codeでは設定のスコープを以下の3段階で管理できます。

| スコープ | 対象 |
|---|---|
| ユーザー | 全プロジェクトで自分が使う |
| プロジェクト | そのプロジェクトで全員が使う |
| ローカル | そのプロジェクトで自分だけが使う |

どのプロジェクトでも利用するような汎用的なMCPはユーザースコープで導入、プロジェクト特有のMCPはプロジェクトスコープで導入といった使い分けをすることで、上記の取捨選択を行います。
例えば、AWSのMCPをユーザースコープで追加してしまうと、AzureやGoogle Cloudを用いているプロジェクトレポジトリでもAWS MCPのtoolsが読み込まれてしまいます。このとき、使われる可能性が低いAWS MCP由来のtoolsは完全に無駄なコンテキスト読み込み・無駄なトークン消費になってしまいます。そのため、AWS MCPはAWSをインフラ基盤として用いているプロジェクトでのみプロジェクトスコープで導入するのが適切です。

プロジェクトに関係のないMCPを無駄に読み込まないよう、スコープは適切に管理しましょう。

## Claude Codeが作業するために自動で開くターミナルに独自の環境変数を渡したいんだけど、できる？

Claude Code作業するときに用いるターミナルは、それまでユーザーが使っていたターミナルセッションをそのまま用いるのではなく、Claude Code自身が自ら新しくターミナルセッションをを立ち上げて作られます。
`bash_profile`や`.zshrc`などのファイルに環境変数の初期セットアップスクリプトが書かれていれば、Claude Codeが立ち上げたターミナルにもそれら設定が反映されますが、機密度の関係で開発者が手動で`export SOME_ENV=xxxx`とやってセットしていた場合にはそれはClaude Codeには引き継がれません。

このように、何らかの都合により`bash_profile`や`.zshrc`に書けない環境変数をClaude Codeのターミナルに渡したい場合は、`.claude/settings.local.json`の`env`フィールドに書いておくことで、プロンプトやユーザーの指示に頼らず確実に値のセットを行うことができます。
```json
{
  "env": {
    "MOMENTO_API_KEY": "XXXXXX"
  }
}
```
このファイルは`.gitignore`に追加して、シークレット情報がリポジトリに混入しないよう注意してください。


## よくやる作業のたびに毎回プロンプトを書くのが面倒になってきた。楽できる方法はある？

「git add → commit → push」のようなよく繰り返される作業を毎回自然言語で指示するのは非効率です。
とはいえこの例の難しい点としては、gitのコミットメッセージは毎回固定ではなくdiffに応じたものをその時々で考えたいとするとなると、決まりきったMakefileや自作bashコマンドのようなやり方では実現不可能だということです。

以上の例のような「AIに考えてよしなにどうにかしてほしい部分が作業の中にはあるのだけど、コマンドとしては定型化したい」という定型作業には**カスタムコマンド**が使えます。

カスタムコマンドは`.claude/commands/`ディレクトリにMarkdownファイルを置くことで作成します。
今回であると例えば`.claude/commands/commit.md`というファイルを作成しておくと、`/commit`とコマンドを打つだけでその内容のプロンプトが実行されます。

```markdown:.claude/commands/commit.md
# /commit カスタムコマンド

このコマンドは、変更をgit commitします。コミットメッセージは引数として受け取ります。

## 手順

1. 現在のgit statusを確認
2. 変更をdiffで表示  
3. 最近のコミット履歴を確認してコミットメッセージのスタイルを把握
4. すべての変更をステージングエリアに追加
5. 引数で受け取ったメッセージでコミットを作成
6. コミット作成後の状態を確認

## 注意事項

- git repositoryでない場合はエラーメッセージを表示
- 変更がない場合は何もしない

## 実行

以下の手順でgit commitを実行してください：

1. `git status` でステータス確認
2. `git diff` で変更内容確認  
3. `git log --oneline -3` で最近のコミット確認
4. `git add .` で変更を追加
5. 引数のコミットメッセージを使って `git commit` 実行
6. `git status` で完了確認
```

## コマンドを人間の私が手動実行するんじゃなくて、Claude Code自身にも作業の中で自発的によしなに使って欲しくなった。どうすればいい？

カスタムコマンドはあくまで「開発者である人間が自分から呼び出す」という使い方をすることしかできません。このコマンドの存在をClade Code自身が察知してよしなに使うということはしません。
このように、「人間だけではなくClaude Codeにも自発的に使ってほしい」というような作業の場合は**Skill**を使って定形化しましょう。

「こういう状況になったらこのSkillを使うべき」という説明をSkillファイルに書いておくことで、Claudeが文脈を読んで自動的に使ってくれます。また、SkillはMarkdownのプロンプトだけでなく、スクリプトやリファレンスドキュメントなどを同ディレクトリに置くこともできます。

```
.claude
└── skills
    └── skill-name
        ├── SKILL.md          # 必須：プロンプト指示
        ├── scripts/          # 任意：中で使うスクリプト
        │   └── script.py
        ├── references/       # 任意：参照ドキュメント
        │   └── doc.md
        └── assets/           # 任意：テンプレート等
            └── template.html
```

これら`scripts/`や`references/`ディレクトリ内のファイルは、`SKILL.md`とは異なりスキル実行中に必要になったときだけ読み込まれます。
コマンド単独ではできないような作り込みをしつつ、なおかつ必要な情報のみを都度Claudeに渡すようになっていることで、コンテキストを節約しながら効率的に複雑かつ柔軟な処理を実行させることができます。

:::message
このような「初めから全てをコンテキストにロードしておくのではなく、必要になったときに都度読み込む」というAnthropics推奨のベストプラクティスのことをProgressive Disclosure(段階的開示)といいます。
:::

Skillはカスタムコマンド同様`/skill-name`と打って手動実行させることもできますが、上述した通りClaudeと自然対話している中で「今はこのSkillを使うべきだ」とClaudeが判断したら自動で発火するという利用方法もあります。
この性質により、「とあるSkill Aを利用してClaudeと会話しているときに、別にSkill Bに処理をPASSした方がいいと判断されてSkill Bが起動する」といったリレーが行われることがあります。
これを狙ってうまく自作Skillを複数作成することで、Claude Codeで実行できるある種のワークフローをも作ることができます。

## こんな作り込まれた複雑なSkillを自力で作るのは大変そう。何か楽できるいい方法ある？

作り込まれたSkillはカスタムコマンドと比べてかなり複雑な構成や多くのプロンプトを必要としますが、anthoropicが「Skillを作るためのSkill」の**skill-creator**を出してくれています。
このskill-creatorを起動して作りたいSkillの内容を伝えるだけで、いい感じに自作のSkillを自動生成してくれます。

```
/plugin install example-skills@anthropic-agent-skills
```
```
$ claude
> XXXXXを普段やっているんだけど、これをSkill化できませんか
→ これだけでskill-creatorが起動していい感じのSkillを作ってくれる
```

## SkillとかMCPとか、諸々のClaude Codeの便利機能を一つ一つ設定するのは大変なんだけど...

Skills、Agents、Hooks、MCPを組み合わせて育てたうまい開発環境を、自分だけではなくチームメンバーにも展開しようとすると大変です。
設定ファイルやプロンプトを一つ一つ共有・説明するのは手間がかかるからです。

そこで使えるのが、Skills・Agents・Hooks・MCPをまとめてパッケージ化し一括で共有できる**Plugin**という仕組みです。
チームのスタンダードな設定セットをPluginとして用意しておけば、新しいメンバーが加わったときもすぐに同じ環境を再現できます。

また、先に紹介したskill-creatorも、Anthropics公式が提供している`example-skills`プラグインの一部として提供されています。
内輪のチームの中での設定共有だけではなく、どこかの企業や他の開発者が公開している設定をインストールして利用するということもプラグインを通して行われています。

## Claude Codeに作業させている間に、自分が別ブランチにチェックアウトして他の作業を並列でやるとかできない？

ここまでClaude Codeの数々の機能を駆使することができるようになると、かなり複雑な設計・実装もClaudeに自律的に任せられるようになっているかと思います。
permissionによる許可セットもうまいこと設定できているのであれば、自分が作業に介入するのは確認するべきコマンド実行の承認や、Plan時にClaudeから聞いてくる少しの質問返しぐらいになっているというケースもあるのではないでしょうか。

そうなると、Claudeの作業を待っている間は人間である私たちは暇だな〜となります。
1つのチケットの実装をClaude Codeに実装させている間、自分は別のチケットの内容をClaude Codeの別セッションでPlanさせて作業したいということもあるでしょう。
ただそうなると、gitの「通常1つのレポジトリには1つの作業ディレクトリしかなく、ブランチを切り替えると作業ディレクトリの内容がそのブランチの内容に入れ替わってしまう」という仕様が足枷となります。

そこで有効活用できるのが、1つのレポジトリで複数の作業ディレクトリを作ることができる**git worktree**の仕組みです。
これを活用して複数のブランチを別ディレクトリとして同時に展開し、それぞれのディレクトリでClaude Codeのセッションを動かすことによって並行作業が可能になります。

```bash
$ git worktree add workspace/feature-a branch-a
$ git worktree add workspace/feature-b branch-b
```
```
.
└── workspace
    ├── feature-a # このディレクトリの中にbranch-aの内容がチェックアウトされて展開されるようになる
    └── feature-b # このディレクトリの中にbranch-bの内容がチェックアウトされて展開されるようになる
```
