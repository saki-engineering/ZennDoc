# この記事について
「とりあえずClaude Codeを入れてみたけど、なんか使いこなせていない気がする」「あの機能の存在は知っているけど、自分には関係ないと思っていた」という方はいませんか？

本記事では、Claude Codeを使いはじめた人が日々の作業の中で感じるよくある「つまずき」や「不満」を出発点に、それを解消するClaude Codeの機能を紹介していきます。
- Claude.aiチャットとClaude Codeのagentモードは何が違うの？
- CLAUDE.md、Hook、MCP、Skillなど設定項目が多すぎてどこから手をつければいいかわからない

という部分を、「こういう不満があるなら、この機能を使いましょう」という形で解説し、Claude Codeを業務に本当に活かせる状態を目指すことを目的としています。

:::message
本記事では各機能の詳細な使い方よりも、「どんな課題を解決するための機能か」という視点での紹介に留めます。詳細な設定方法はClaude Code公式ドキュメントをご参照ください。
:::

# Claude Code 使いこなしQ&A集

## Claude.aiのチャットだけでは不十分かな？
Claude.aiのWebサイトやアプリを開いて、「このコードを直して」「この仕様を調べて」と指示を出す——そんな使い方をしている方は多いのではないでしょうか。それでも十分便利ですが、Claude.aiのチャットはあくまで「会話」が中心です。実際にファイルを読み込み、コードを書き、コマンドを実行するといった「作業」の部分は、人間が手を動かす必要があります。

Claude Codeのagentモードでは、その作業まで丸ごと任せることができます。「このissueを解決して」と一言伝えるだけで、関連ファイルを読み込み、修正し、テストまで実行してくれます。チャットが「アドバイスをもらう場」なら、agentモードは「そのまま作業を代行してもらう場」というイメージです。


## 毎回同じ指示を出している気がする...チーム固有のルールを覚えてほしいんだけど、できる？

「うちのプロジェクトではコミットメッセージは英語で書く」「テストコマンドは`make test`」——こういったローカルルールを毎回会話の冒頭で説明するのは非効率です。そこで使えるのが`CLAUDE.md`です。プロジェクトのルートに置いたこのファイルは、Claude Codeがセッション開始時に必ず読み込みます。テストコマンドやブランチ命名規則、コーディングスタイルなど、プロジェクト固有のルールをここに書いておくことで、毎回説明する手間が省けます。

```markdown
## テスト
- テストコマンド: `make test`

## ブランチ命名ルール
- feature/[issue番号]-[概要]

## コミット
- メッセージは英語で書く
```

このファイルはGitでコミットしておけばチームメンバー全員が恩恵を受けられます。


## そのチーム固有のルールとやらをわざわざあ言語化するのもめんどくさいズボラなんですけど、なんとかなりますか？

CLAUDE.mdに書くルールを一から考えて言語化するのが面倒、という方もいるでしょう。そんなときは`/init`コマンドが使えます。

Claude Codeが現在のリポジトリを読み込み、コードスタイルや既存のMakefileのターゲット、package.jsonのスクリプトなどを解析して、CLAUDE.mdの雛形を自動生成してくれます。生成されたものを確認・調整するだけでよいので、白紙から書くよりずっと楽です。


## Claude Codeが作業をするたびに、一挙手一投足いちいち許可を求められるのが煩わしいんだけど...

Claude Codeがファイルを読んだり、コマンドを実行したりするたびに「許可しますか？」と聞いてくる——慣れるまでは安心感がありますが、作業を繰り返すうちにこの確認が煩わしくなってきます。

許可を自動化したい操作は、`settings.json`の`permission`に書いておくことで、いちいち確認なしに実行させることができます。WebSearchやWebFetchといった情報収集系の操作は許可しておいても問題ない場面が多いでしょう。

## Claude.mdに書いたルールを最近Claudeが守ってくれなくなっちゃったのだけどどう躾けたらいいですか？
CLAUDE.mdにフォーマッターの適用を書いたのに、Claudeが書いたコードにフォーマッターがかかっていないことがあります。

これはプロンプトが「お願い」であることに起因します。Claudeはコンテキストや状況に応じてプロンプトの解釈を変えることがあり、100%の保証はありません。強制的に実行させたいものには**Hook**を使いましょう。Hookは特定の操作（ファイル編集後、コマンド実行前など）に紐づけてシェルスクリプトを実行する仕組みです。「Claudeがファイルを編集したら必ずフォーマッターを実行する」といった処理を、会話ではなくシステムとして保証できます。

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "gofmt -w $CLAUDE_TOOL_RESULT_FILE_PATH"
          }
        ]
      }
    ]
  }
}
```


## Claude Codeに作業を任せて放置しておいたら、処理実行許可を求めるだいぶ序盤のところで止まってました。やっぱり完全放置はできなくてずっと画面を見張ってないといけないんですか？

AgentモードでClaudeに作業を任せておいて、後から見たら入力待ち状態で止まっていた——という経験はないでしょうか。これも**Hook**で解決できます。「ユーザーの入力が必要な状態になったら音を鳴らす」という処理をHookで登録しておけば、離席中でも気づくことができます。

`/hook`コマンドを使うと、対話形式でHookの設定を作成できます。会話でのプロンプトと違い、Hookは**特定のイベントが発生したときに100%実行される**のが最大の特徴です。プロンプトは確率的な「お願い」ですが、Hookは確実な「仕組み」です。

## 前の会話に引きずられておかしな動きをし出したんだけど....

作業を続けていくと、前のセッションで失敗したアプローチがコンテキストに残り、同じ間違いを繰り返したり、的外れな修正を始めたりすることがあります。Claudeが間違える→修正する→また間違える、というループに陥ったとき、それはコンテキストが失敗したアプローチで汚染されているサインです。

こういうときは躊躇せず`/clear`で会話履歴をリセットするか、新しいセッションを開始しましょう。状態のリセットを怖く感じるかもしれませんが、Claude Codeはコードとして成果物が残ります。会話履歴がなくなっても、成果物は消えません。


## 一つのセッションで話したいことが多すぎてそれだけでコンテキストが溢れてしまうんだけど、どうやって会話を継続すればいい？

`/clear`は使いたくないけど、会話が長くなってコンテキストウィンドウの上限に近づいてきた——そういうときは`/compact`コマンドが使えます。

`/compact`は会話履歴を要約してコンパクトにまとめ直します。全ての詳細を保持するわけではありませんが、作業の文脈は維持しながらコンテキスト使用量を減らすことができます。`/clear`より穏やかなリセット手段として活用できます。


## これから色々試行錯誤をしてコンテキストを汚すからその前に今の状態を保存して後で戻れるようにしたいんだけど、一時セーブとかできる？

「ここから実験的なアプローチを試してみたいけど、失敗したら今の状態に戻りたい」というときには、**rewind**と**fork**が使えます。

`rewind`は直近1つの操作を元に戻します。小さな一歩を巻き戻したいときに使います。`fork`は、現在のセッションを「親」として、その続きを引き継いだ新しいセッションを作ります。元のセッションには手を加えないため、フォークしたセッションで失敗しても、親セッションがそのまま残っています。gitのブランチのセッション版と思えばわかりやすいかもしれません。


## Claude Codeにちょっと大きい設計・実装作業を任せてるのだけど、Agentモードで逐一ペアプロしていると軌道修正ばかりでなかなかゴールに辿りつかない...

Agentモードで作業を任せていると、「今Claudeが何をしようとしているのか」「最終的にどこに着地するのか」が見えないまま、どんどん変更が加えられてしまうことがあります。1行ずつの変更だけ見ていても、全体像が把握できないのです。最初に全体像を見せてほしい、という気持ちになる場面があるかと思います。

**Planモード**はその問題を解決します。実装に入る前にClaudeが「こういう方針で進めます」という計画を提示してくれるので、人間がそれを確認してからGOを出すことができます。また、Planで承認した後の実装フェーズでは、個別の操作ごとに許可を求めることなく一気に作業を進めてくれます。全体像を確認してから、実装は高速に——という流れが実現できます。


## 大きな作業をしていると、1つのセッションで使えるコンテキストがすぐに枯渇してしまうようになった。何かいいコンテキスト節約方法はある？

いろいろな作業をClaudeに任せているうちに、コンテキストウィンドウがすぐに一杯になってしまうことがあります。そんなときは**SubAgent**を活用しましょう。

例えば「最新のライブラリの使い方を調べてきて」というリサーチ作業は、SubAgentとして別セッションで動かすことができます。人間が手動で別ウィンドウを開いてリサーチして、結果をコピーして元のセッションに貼り付ける——その一連の作業ごとSubAgentに任せられるのが魅力です。試行錯誤の過程はSubAgentに押し付けて、最終的な結論だけ元のセッションに持ち帰らせる、という使い方もコンテキスト節約に効果的です。

`/agent`コマンドで作成できます。やりたいことの説明（description）を簡単に書けば、使用するツールの設定などは質問に答えるだけで構築してくれます。


## SubAgentを使ってもまだコンテキストが足りない...

ここまで使いこなしているなら、CLAUDE.mdが肥大化してきていないでしょうか。プロジェクトのREADMEと見分けがつかないくらい長くなっているなら、**rules**の活用を検討しましょう。

rulesは、特定のパスのファイルを扱うときにだけ読み込まれるルールを定義できます。「このディレクトリのファイルを触るときだけ、この追加ルールを読む」という設定が可能なため、CLAUDE.mdに全て書き込むよりもコンテキストを節約できます。

:::message
CLAUDE.md内で`@file_name`と書くことで、詳細な内容を別ファイルに逃すことができます。ただし、これはCLAUDE.md参照のたびに読み込まれてしまうため、コンテキストの節約効果はありません。コンテキスト節約が目的であればrulesを使いましょう。

参考: https://code.claude.com/docs/ja/memory
:::


## エディタでファイルをいじる・ターミナルでコマンドを実行する以外の操作もClaudeに任せたいけど、できる？

gitのpushやPRの作成など、エディタの外での操作もClaudeに任せたいという場面が出てきます。そういう場合は**MCP（Model Context Protocol）**を活用しましょう。MCPとは、外部ツールやサービスとClaude Codeを繋ぐインターフェースです。GitHub MCPを使えば、PRの作成やブランチの管理をClaudeに任せることができます。

```bash
claude mcp add github -e GITHUB_PERSONAL_ACCESS_TOKEN=XXXXX -- github-mcp-server stdio
```

Claude Codeに追加したMCPの設定は`~/.claude.json`に記録されます（Claude Desktopの設定とは別管理です）。

:::message
[MCP is dead. Long live the CLI](https://ejholmes.github.io/2026/02/28/mcp-is-dead-long-live-the-cli.html)という記事が出ているように、「そのままCLIで実行すればよいのでは」という場面もあります。CLIで十分な操作はCLIに任せるほうがトークン消費も少なく合理的です。CLIで困らないものはCLIを使う、用途に応じて選択していくのがよいかと思います。
:::


## Claude CodeがWeb検索・WebFetchするときのやり方が非効率な気がする。何かいいチューニング方法はある？

MCPはツールを操作するだけでなく、情報を参照するためにも使えます。例えば**context7**というMCPを使うと、ライブラリの公式ドキュメントを高精度で参照できます。Web検索は広く浅く情報を拾いますが、context7のようなドキュメント特化のMCPはより正確な情報をClaudeに渡せます。AWSも多数のMCPを公式提供しており、AWSサービスの最新情報を参照するのに便利です。


## MCPを増やしたら作業の精度が下がった気がする...

MCPを大量に追加していくと、Claudeが「どのツールを使えばいいか」判断に迷うようになります。これがかえって精度低下につながることがあります。

Claude Codeでは、MCPのスコープを3段階で管理できます。

| スコープ | 対象 |
|---|---|
| ユーザー | 全プロジェクトで自分が使う |
| プロジェクト | そのプロジェクトで全員が使う |
| ローカル | そのプロジェクトで自分だけが使う |

優先度はローカル＞プロジェクト＞ユーザーの順です。AWSのMCPをユーザースコープで追加してしまうと、AzureやGCPのプロジェクトでも読み込まれてしまいます。プロジェクトに関係のないMCPを無駄に読み込まないよう、スコープは適切に管理しましょう。


## Claude Codeが作業するために自動で開くターミナルに独自の環境変数を渡したいんだけど、できる？

Claude Codeが別プロセスで作業するとき、自分のターミナルで設定していた環境変数が引き継がれないことがあります。`bash_profile`や`.zshrc`に書けないシークレット情報などは、`.claude/settings.local.json`の`env`フィールドに書いておきましょう。

```json
{
  "env": {
    "MOMENTO_API_KEY": "XXXXXX"
  }
}
```

このファイルは`.gitignore`に追加してシークレット情報がリポジトリに混入しないよう注意してください。


## よくやる作業のたびに毎回プロンプトを書くのが面倒になってきた。楽できる方法はある？

「git add → commit → push」のような繰り返す作業を毎回自然言語で指示するのは非効率です。コミットメッセージのような「AIに考えてほしいけど、コマンドとしては定型化したい」作業には**カスタムコマンド**が使えます。

`.claude/commands/commit.md`のように、`.claude/commands/`ディレクトリにMarkdownファイルを作成しておくと、`/commit`とコマンドを打つだけでその内容のプロンプトが実行されます。コミットメッセージを自動生成してpushするまでの一連の流れを定義しておけば、`/commit`一発で済むようになります。


## コマンドを人間の私が手動実行するんじゃなくて、Claude Codeが作業してくれている中でも自発的によしなに使って欲しくなった。どうすればいい？

カスタムコマンドはあくまで「自分から呼び出す」ものです。「Claudeが自発的に使ってほしい」という場合は**Skill**を使いましょう。

SkillはコマンドとしてもClaudeが自律的に使用することもできます。「こういう状況になったらこのSkillを使うべき」という説明をSkillファイルに書いておくことで、Claudeが文脈を読んで自動的に使ってくれます。また、SkillはMarkdownのプロンプトだけでなく、スクリプトやリファレンスドキュメントなどを同ディレクトリに置くこともできます。

```
.
├── SKILL.md          # 必須：プロンプト指示
├── scripts/          # 任意：中で使うスクリプト
│   └── script.py
├── references/       # 任意：参照ドキュメント
│   └── doc.md
└── assets/           # 任意：テンプレート等
    └── template.html
```

`scripts/`や`references/`のファイルは必要なときだけ読み込まれます（Progressive Disclosure）。コンテキストを節約しながら、必要な情報だけClaudeに渡せるため効率的です。

参考: https://platform.claude.com/docs/en/agents-and-tools/agent-skills/best-practices


## Skillのディレクトリ構成を自力で作るのは大変そう。何か楽できるいい方法ある？

Skillはカスタムコマンドより複雑な構造になりますが、公式Skillの**skill-creator**を使えばゼロから作る必要はありません。作りたいSkillの内容を伝えるだけで、質問に答えながらSkillファイルを自動生成してくれます。


## SkillとかMCPとか、諸々のClaude Codeの便利機能を一つ一つ設定するのは大変なんだけど...

Skills、Agents、Hooks、MCPを個人で整えてきたとして、それをチームメンバーにも展開しようとすると大変です。設定ファイルを一つひとつ共有・説明するのは手間がかかります。そこで使えるのが**Plugin**です。

PluginはSkills・Agents・Hooks・MCPをまとめてパッケージ化し、一括で共有できる仕組みです。チームのスタンダードな設定セットをPluginとして用意しておけば、新しいメンバーが加わったときもすぐに同じ環境を再現できます。


## Claude Codeに作業させている間に、自分が別ブランチにチェックアウトして他の作業を並列でやるとかできない？

Agentに長い作業を任せているとき、その完了を待たなければ次の作業に入れない——そう思いがちですが、**git worktree**を活用することで並行作業が可能になります。

git worktreeを使うと、同じリポジトリで複数のブランチを別ディレクトリとして同時に展開できます。Agentがブランチ1で作業している間に、自分（またはもう一つのAgentセッション）がブランチ2で別の作業を進められます。チケットが山積みでも、Agentを並走させることでスループットを上げることができます。


# まとめ

冒頭の問いについての解答をまとめます。

- Claude.aiチャットとClaude Codeのagentモードは何が違うの？
    - チャットは「アドバイスをもらう場」、agentモードは「作業を代行してもらう場」。ファイルの読み書きやコマンド実行まで自律的にやってもらえる。
- CLAUDE.md、Hook、MCP、Skillなど設定項目が多すぎてどこから手をつければいいかわからない
    - 「困ったことがあったら、その課題を解決する機能を使う」で大丈夫。本記事のように「こういう不満 → この機能」でひとつずつ導入していけばよい。

Claude Codeは機能が多い分、全部知らなくても使いはじめられます。まずagentモードで作業を任せてみて、不満を感じるたびに機能を足していくのが、無理なく使いこなす近道ではないでしょうか。
