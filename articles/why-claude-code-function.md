## MCPをいっぱい入れたら精度悪くなった気がする。
MCPをいっぱい入れると、その中のどれを入れようかと迷ってしまうことがある。
だから、スコープを絞るといい。

Claude Codeでは3つのスコープが用意されている
1. ユーザー: 全プロジェクトで自分が使う
2. プロジェクト: そのプロジェクトで全員が使う
3. ローカル: そのプロジェクトで自分だけが使う

3の方が優先度が上です。ユーザーに何でもかんでも入れていると、関係ないプロジェクトでも使われてしまう。
例えば、AWSのMCPをユーザーで入れていたら、AzureやGoogle Cloudのプロジェクトでも無意味に使われてしまい効率が悪い。

## Claude Codeが作業するときに別のターミナルが開いちゃう。個別に設定していた環境変数が反映できない。
bash_profileなどにかけないものは、`.claude/settings.local.json`に書けばいい
```json
{
  "env": {
	"MOMENTO_API_KEY": "XXXXXX"
  }
}
```

## いちいち指示を自然文で書くのすら最近めんどくさいのだが。
commandsを作りましょう。
例えば編集した内容をgit commit→pushしたい場合。コミットメッセージが決まっているなら`git add . & git commit -m "xxxxx" & git push`とすればいいんだけど、コミットメッセージは流石に今のstageの内容をよしなにサマリして書いて欲しい。だからMakefileとかでカスタムコマンドを作るとかもできない。
そうなった時はclaudeでコマンドを作る。

`.claude/commands/commit.md`とかでマークダウンを作る。

こうしておくと、`/commit`とするだけで、プロンプトの処理が実行される。tool実行とか人が主体的にやる操作は、このコマンドで定義するのがいい。

## commandだと結局自発的にしか使えないよね
そういう場合はskillsを使おう
skillsだと、コマンドみたいに/で使うのもできるし、普通に自然文で指示している中でClaudeが勝手に使ってくれることもできる

skillだと、マークダウンだけじゃなくてスクリプトとかも置ける。複雑なことをやりやすい。
```
.
├── SKILL.md          # 必須：プロンプト指示
├── scripts/          # 任意：中で使うスクリプト
│   └── script.py
├── references/       # 任意：参照ドキュメント
│   └── doc.md
└── assets/           # 任意：テンプレート等
    └── template.html
```
任意のところは、必要なときにしか読まれないからコンテキストの節約にもなる。Progressive Disclosureという考え方。

https://platform.claude.com/docs/en/agents-and-tools/agent-skills/best-practices

## Skillの中のディレクトリ構成が複雑で、こんなの自力で作る気にならないのだが。commandならさておき。
skills-createrという公式Skillがあるよ。それを使えば、質問に答えるだけで勝手によしなに作ってくれる。
